The one equation to remember

So what actually is an agent?

Agent = Model + Harness

A raw model just turns text in → text out. It becomes an agent the moment a harness wraps it with state, tools, feedback loops, and constraints.

The Model

The intelligence. Takes in text / images / audio and outputs text. Brilliant, but inert on its own.

The Harness

"Every piece of code, configuration, and execution logic that isn't the model." It makes the intelligence useful.

Framing: LangChain — "The Anatomy of an Agent Harness"

Concepts worth knowing
Reasoning / "thinking" modelsspend extra compute to reason before answering — better at multi-step agent work
Context windowhow much it can read at once — the scarce resource a harness must manage
Tool use / function callingemits structured calls — the capability that makes agents possible at all
Tokens · input vs outputthe unit of pricing — input (what you send) is cheap; output (what it writes) is typically 4–5× more; long context bloats the input bill
KV cache / prompt cachingreuse the model's prefix compute across calls — big latency & cost win when system prompts & tool defs stay stable (agents re-send the same prefix every turn)

How we got here

Three eras of "talking to models"

Each era didn't replace the last — it wrapped it. Today's harnesses are mostly delivery mechanisms for good context engineering.

P
~2020–2022

Prompt engineering

Craft the perfect single message. "Think step by step," few-shot examples, role-play. The model is a stateless oracle.

C
~2023–2024

Context engineering

The prompt is dynamic. Retrieve the right files, memories, and tool results — and manage the window as a scarce resource.

H
2025 →

Harness engineering

Build the whole system around the model: loops, tools, subagents, hooks, compaction. "If you're not the model, you're the harness."

🧠

Context is precious & scarce. Harnesses add strategies like compaction (summarize & offload) and continuation (re-inject the goal into a clean window) to keep working past the limit.

Deep dive · Claude Code

Six building blocks of a real harness

🔌MCP
Tools / hands. An open protocol that plugs the agent into live systems — GitHub, Postgres, Sentry, Slack. No more pasting logs; it reads & writes the real thing. claude mcp add ...
📖Skills
Knowledge / playbooks. A SKILL.md with a short description that auto-loads only when relevant (progressive disclosure) — reusable procedures that cost ~0 context until triggered.
🤖Subagents
Delegation & isolation. Spin up a worker with its own context window for noisy side-tasks (e.g. searching the whole repo); only the summary returns. Fan out many in parallel.
⌘Commands
Control surface. What you type to drive a session — /model, /compact, /plan, or your own saved skills like /code-review.
🪝Hooks
Deterministic guardrails. Shell/HTTP automations on lifecycle events (PreToolUse, PostToolUse…) — auto-format on edit, block an rm -rf. Outside the model's discretion.
⚡Routinesnew
Autopilot ("workflows"). A saved prompt + repos + connectors that runs unattended in the cloud — on a schedule, a GitHub event, or an API call. Composes all the layers above.
🧭

Mental model: MCP = tools · Skills = knowledge · Subagents = delegation · Commands = how you invoke · Hooks = guardrails · Routines = scheduled autonomy.

The AI bar & the AI ceiling

A cheap first ask already gets you a strong answer — that's the AI bar. The remaining climb to the AI ceiling is on you: add context, turn by turn.

Two ideas worth knowing

RAG, and how people build agents

Concept

RAG — Retrieval-Augmented Generation

Don't fine-tune facts in — fetch them at query time. Embed your docs/codebase into vectors, retrieve the top-k relevant chunks, and paste them into the prompt as context.

🔎

For code, the field is shifting from vector-RAG → agentic search: instead of pre-indexing, the agent just greps & reads files on demand (how Claude Code works). Often more accurate for repos.

Debate

Framework vs. model-native

Framework-style

LangChain · LangGraph
  • Explicit graphs & orchestration you control
  • Great when steps must be reliable / auditable
  • More scaffolding to maintain

Model-native

Claude Agent SDK · OpenAI Agents SDK
  • Trust a strong model in a simple tool loop
  • Less code; the model does the planning
  • Rides the curve as models improve
📈

The trend (per Anthropic's Building Effective Agents): start with direct API calls; add a framework only when it demonstrably helps. As models improve, harnesses get thinner — more logic moves into the model.
